# 后续迭代 TODO（任务备忘）

> 记录与 **OpenClaw** 分工、出站动作、以及本仓库 RAG 边界，便于后续迭代对齐。

---

## 背景结论（摘要）

- **业务流程类能力**（如：年度报告提交排查 → **通知未提交人员**、对接 OA/HR/IM）更适合放在 **OpenClaw（或同类工具/编排层）**：集中管理 **凭证、模板、审计、人工确认、出站频率**，避免 RAG 服务承担过多集成与合规风险。
- **本仓库** 聚焦：**检索与反幻觉**、**确定性 What-If 工具**（如 `POST /api/whatif/loan/compare`）、**只读、可单测** 的能力。
- **协作方式**：两边通过 **清晰 HTTP/API 契约** 衔接；RAG 可提供 **有据可查的摘要/试算**；**名单与「谁该被通知」以权威业务系统为准**，宜在 OpenClaw 侧拉取或经审批后再发。

---

## 职责边界（备忘）

| 维度 | 本仓库（MAP-RAG / DynaSense） | OpenClaw（或同类编排层） |
|------|-------------------------------|---------------------------|
| 核心职责 | 知识检索、混合路由、grader/generate、citations、锚点过滤、What-If **数学/确定性** API | **动作**：查状态、发提醒、工单、审批流、与 IM/邮件集成 |
| 凭证与出站风险 | 尽量少持有「可代表用户群发」的密钥；避免在 `/demo` 内嵌高风险写操作 | 更适合托管 **Bot 账号、模板、审计日志、幂等与重试** |
| 演进 | 避免被 HR/OA/通知渠道绑死，保持 **可测、可回滚** | 连接器独立迭代，不拖垮 RAG 发版节奏 |
