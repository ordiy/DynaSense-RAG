# 企业级 RAG 架构演进与风险控制 (基于 MVP 验证)

> **核心原则：在企业环境下，给出一个坏的/有毒的答案，比没有答案问题更大。** (No Answer is better than a Bad/Toxic Answer.)

## 1. 对 Query Rewrite 的反思与放弃
在 MVP 验证阶段，我们明确拒绝了在主链路上默认开启 **Query Rewrite (实时查询重写)**。
*   **黑话灾难与意图漂移**：企业内部充斥着专有名词和缩写，通用 LLM 的重写往往会“抹平”这些关键特征，导致南辕北辙。
*   **延迟成本**：每次查询增加 1-3 秒的 LLM 生成延迟，破坏了检索系统的核心体验。
*   **运维黑盒**：引入了不可预测的不可控变量，当业务方质疑检索结果时，无法进行确定性的追溯。

## 2. “宁缺毋滥”的设计哲学：防幻觉体系 (Anti-Hallucination)
在企业知识库、财务合规、人力政策等场景中，大模型的“一本正经的胡说八道”是绝对不可接受的红线。基于这一原则，MAP-RAG 架构的设计重心必须向**阻断机制**倾斜：

### 2.1 强硬的认知层防波堤 (Grader Node)
在我们的 LangGraph 节点设计中，`grade_documents_node` 扮演着至关重要的角色。它的首要任务**不是**拼凑出答案，而是：
*   **严格比对**：判断检索召回的 Document 是否**实质性地**包含了解答 Query 的事实依据。
*   **断然拒绝**：如果判断为 `no`，系统必须有勇气直接阻断生成流程，返回“上下文中未找到相关信息”的标准化话术。不能任由生成节点 (Generator) 基于大模型的内置参数“猜”一个答案。

### 2.2 溯源与可解释性 (Citation & Traceability)
每一句由 LLM 吐出的回答，必须能精确映射到具体的文档源（Document ID / 原文摘录）。如果一句话无法挂载引用，它在企业级 RAG 中就是“非法产物”。这在架构上要求在 Prompt 中强约束 LLM 的输出格式，严格绑定参考资料。

## 3. 企业级检索体验的正确演进路径
既然拒绝了实时 Query Rewrite，为了弥合“用户口语”与“企业公文”的检索鸿沟，架构应向以下方向演进：

*   **算力左移 (Doc2Query / HyDE 在入库端应用)**：在文档入库 (Ingestion) 阶段，离线调用 LLM 预测该文档可能回答的问题，将这些预测问题作为 Metadata 或附属文本一并向量化。这既提升了检索准度，又保证了查询时的零额外延迟。
*   **多路混合召回 (Hybrid Search)**：坚持保留用户的原始查询。采用 Dense Retrieval (Embedding) 负责捕获模糊意图，结合 Sparse Retrieval (BM25/Elasticsearch) 确保精准词汇（如特定项目代号、员工工号）的绝对命中。最后通过 RRF (Reciprocal Rank Fusion) 算法合并排序。
*   **精准的意图路由 (Metadata Filtering)**：与其重写问题，不如识别条件。利用轻量级模型快速提取查询中的时间、部门、文档类型等实体，将其转化为数据库的硬性过滤条件 (Filter)，从物理层面缩小检索空间，提高准确率。
